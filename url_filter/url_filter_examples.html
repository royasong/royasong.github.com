1. 기본 유효성 검사 (STEP 1)

설정과 무관하게 URL 자체 속성으로 바로 결과가 결정되는 케이스입니다.
Host URL결과판단 근거
not-a-url-$$%% kUrlBlocked STEP1 gurl.is_valid() == false
javascript:alert(1) kUrlBlocked STEP1 gurl.is_valid() == false (javascript: 스킴 무효)
https://myapp.example.com/ kUrlAllowed STEP1 host_url_ = "https://myapp.example.com/" 과 완전 일치
about:blank kUrlAllowed STEP1 SchemeIsHTTPOrHTTPS() == false → 필터링 제외
file:///sdcard/index.html kUrlAllowed STEP1 file: 스킴 → 필터링 제외
blob:https://example.com/uuid kUrlAllowed STEP1 blob: 스킴 → 필터링 제외
https://example.com/chrome-distiller/article kUrlAllowed STEP1 kAlwaysAllowed: "chrome-distiller" 포함

2. Unsupported 도메인 — 루트만 차단

패턴 목록이 빈 배열([])인 경우, 루트(/) 접근만 차단하고 하위 경로는 통과시킵니다.
// JSON 설정 { "unsupportedList": { "naver.com": [] // 빈 배열 → 루트("/")만 차단 } }
Host URL결과판단 근거
https://naver.com/ kUrlNotSupported DomainIs("naver.com") ✓ · 패턴 없음 · path = "/" → 루트 차단
https://naver.com kUrlNotSupported path 비어있음(empty) → 루트로 간주
https://www.naver.com/ kUrlNotSupported DomainIs("naver.com") ✓ (서브도메인 포함) · 루트 차단
https://m.naver.com/ kUrlNotSupported DomainIs("naver.com") ✓ · 루트 차단
https://naver.com/news kUrlAllowed path = "/news" → 루트 아님 → continue → 이후 black/white 체크 (규칙 없으면 ALLOWED)
https://naver.com/news/article?id=1 kUrlAllowed 하위 경로 → unsupported 통과 → 이후 규칙으로 진행
https://kakao.com/ kUrlAllowed DomainIs("naver.com") ✗ → unsupported 미해당 · 이후 규칙 없으면 ALLOWED

3. Unsupported 도메인 — 특정 경로 차단

패턴 목록이 있으면 MatchesCompiledPatterns()로 경로를 비교합니다.
// JSON 설정 { "unsupportedList": { "kakao.com": ["/talk", "=/pay"] // "/talk" → kPathPrefix: /talk으로 시작하는 경로 // "=/pay" → kExactPath: /pay 정확히 일치 } }
Host URL결과판단 근거
https://kakao.com/talk kUrlNotSupported kPathPrefix path="/talk" → "/talk"으로 시작 ✓
https://kakao.com/talk/channel kUrlNotSupported kPathPrefix path="/talk/channel" → "/talk"으로 시작 ✓
https://kakao.com/pay kUrlNotSupported kExactPath path="/pay" 정확히 일치 ✓
https://kakao.com/pay/history kUrlAllowed kExactPath path="/pay/history" ≠ "/pay" → 불일치 · ALLOWED
https://kakao.com/ kUrlAllowed 패턴 목록 있음 → MatchesCompiledPatterns 실행 → 루트는 어떤 패턴도 불일치 → ALLOWED
https://kakao.com/map kUrlAllowed "/map"은 어떤 패턴과도 불일치 → ALLOWED

4. Blacklist — 루트 차단

blacklist에 빈 배열로 등록하면 루트 접근만 차단합니다. Unsupported와 동일한 루트 판단 로직입니다.
// JSON 설정 { "blackList": { "daum.net": [] } }
Host URL결과판단 근거
https://daum.net/ kUrlBlocked Blacklist DomainIs ✓ · 패턴 없음 · path="/" → 루트 차단
https://daum.net kUrlBlocked path 비어있음 → 루트 차단
https://m.daum.net/ kUrlBlocked DomainIs("daum.net") ✓ 서브도메인 포함
https://daum.net/news kUrlAllowed path="/news" → 루트 아님 → continue · whitelist 없으면 ALLOWED

5. Blacklist — 경로 패턴 차단

다양한 패턴 타입으로 특정 경로를 차단합니다.
// JSON 설정 { "blackList": { "example.com": [ "/adult", // kPathPrefix : /adult 로 시작하는 모든 경로 "=/restricted", // kExactPath : /restricted 정확히 일치 "*tracker", // kWildcard : path에 "tracker" 포함 "ad-cdn" // kBareSubstring: 전체 URL spec에 "ad-cdn" 포함 ] } }
Host URL결과매칭 패턴판단 근거
https://example.com/adult kUrlBlocked kPathPrefix path="/adult" → "/adult"로 시작 ✓
https://example.com/adult/video kUrlBlocked kPathPrefix path="/adult/video" → "/adult"로 시작 ✓
https://example.com/restricted kUrlBlocked kExactPath path="/restricted" 정확히 일치 ✓
https://example.com/restricted/sub kUrlAllowed kExactPath path="/restricted/sub" ≠ "/restricted" → 불일치
https://example.com/img/tracker.png kUrlBlocked kWildcard path="/img/tracker.png" → "tracker" 포함 ✓
https://example.com/news/tracker-data kUrlBlocked kWildcard path에 "tracker" 포함 ✓
https://ad-cdn.example.com/script.js kUrlBlocked kBareSubstring spec 전체에 "ad-cdn" 포함 ✓ (host에 있어도 매칭)
https://example.com/page?src=ad-cdn kUrlBlocked kBareSubstring query string에 "ad-cdn" 포함 ✓
https://example.com/news kUrlAllowed — 어떤 패턴도 불일치 → whitelist 없으면 ALLOWED

6. Whitelist 단독 (Blacklist 없음)

Whitelist만 있으면 등록된 도메인/경로만 허용되고 나머지는 모두 차단됩니다.
// JSON 설정 { "whiteList": { "naver.com": ["/news", "/sports"], "google.com": [] // google.com: [] → 패턴 없음 → MatchesCompiledPatterns(patterns.empty()) = true → 모든 경로 허용 } }
Host URL결과판단 근거
https://naver.com/news kUrlAllowed DomainIs("naver.com") ✓ · kPathPrefix "/news" 매칭 ✓
https://naver.com/news/article kUrlAllowed kPathPrefix "/news"로 시작 ✓
https://naver.com/sports/baseball kUrlAllowed kPathPrefix "/sports"로 시작 ✓
https://naver.com/ kUrlBlocked DomainIs ✓ 이나 "/", "/news", "/sports" 모두 불일치 → Whitelist miss
https://naver.com/map kUrlBlocked "/map"은 whitelist 패턴과 불일치 → BLOCKED
https://google.com/search?q=test kUrlAllowed DomainIs("google.com") ✓ · 패턴 없음(empty) → patterns.empty() = true → 모든 경로 허용
https://google.com/maps kUrlAllowed google.com 패턴 없음 → 전체 허용
https://daum.net/ kUrlBlocked DomainIs 불일치 → Whitelist miss → BLOCKED

7. Blacklist + Whitelist 조합

Blacklist에서 먼저 차단 여부를 확인하고, 차단 안 되면 Whitelist로 허용 여부를 확인합니다.
Blacklist miss + Whitelist miss = BLOCKED입니다.
// JSON 설정 { "blackList": { "naver.com": ["/adult"] }, "whiteList": { "naver.com": ["/news", "/sports"] } }
Host URL결과경로
https://naver.com/adult/video kUrlBlocked Blacklist "/adult" prefix 매칭 → BLOCKED (whitelist 체크 안 함)
https://naver.com/news/article kUrlAllowed Blacklist miss → Whitelist "/news" 매칭 → ALLOWED
https://naver.com/sports kUrlAllowed Blacklist miss → Whitelist "/sports" 매칭 → ALLOWED
https://naver.com/map kUrlBlocked Blacklist miss → Whitelist miss ("/map" 미등록) → BLOCKED
https://naver.com/ kUrlBlocked Blacklist miss (루트는 "/adult" 불일치) → Whitelist miss → BLOCKED
https://kakao.com/news kUrlBlocked Blacklist DomainIs ✗ → Whitelist DomainIs ✗ → BLOCKED

8. 패턴 타입 5종 비교

동일 도메인 example.com에 각 패턴 타입을 적용했을 때의 매칭 결과입니다.
// 각 타입별 raw 패턴 예시 "example.com": [ "@https://example.com/prefix-url", // kUrlPrefix "=/exact", // kExactPath "*keyword", // kWildcard "/path-prefix", // kPathPrefix "bare-sub" // kBareSubstring ]
타입raw 패턴URL 예시결과이유
kUrlPrefix @https://example.com/prefix-url https://example.com/prefix-url/page 매칭 spec이 "https://example.com/prefix-url"로 시작 ✓
kUrlPrefix @https://example.com/ https://sub.example.com/page 매칭 루트 패턴 → subdomain_host="example.com" → DomainIs ✓ (서브도메인 fallback)
kExactPath =/exact https://example.com/exact 매칭 path="/exact" 정확히 일치 ✓
kExactPath =/exact https://example.com/exact/sub 불일치 path="/exact/sub" ≠ "/exact"
kWildcard *keyword https://example.com/any/keyword/here 매칭 path 어딘가에 "keyword" 포함 ✓
kWildcard * (value 없음) https://example.com/anything 매칭 value 비어있음 → 모든 path 매칭
kPathPrefix /path-prefix https://example.com/path-prefix/deep/page 매칭 path가 "/path-prefix"로 시작 ✓
kPathPrefix /path-prefix https://example.com/other 불일치 path="/other" → "/path-prefix"로 시작 안 함
kBareSubstring bare-sub https://bare-sub.example.com/page 매칭 spec(전체 URL)에 "bare-sub" 포함 ✓ (host에 있어도 OK)
kBareSubstring bare-sub https://example.com/page?ref=bare-sub 매칭 query string에 "bare-sub" 포함 ✓

10. 서브도메인 DomainIs 매칭

DomainIs("example.com")는 host의 뒤쪽이 일치하고, 더 길면 바로 앞에 .이 있어야 합니다.
// 등록 도메인: "example.com"
URL hostDomainIs("example.com")이유
example.com true 완전 일치
www.example.com true suffix 일치 + 앞에 "." 있음
m.example.com true suffix 일치 + 앞에 "." 있음
deep.sub.example.com true suffix 일치 + 앞에 "." 있음
notexample.com false suffix 일치하나 앞에 "." 없음 → 다른 도메인
example.com.evil.net false suffix 불일치 (끝이 "evil.net")
other.net false suffix 불일치

11. 실제 사례 — MK 오디오카 (MkAudioCarTest)

매일경제 오디오카 서비스에 적용된 실제 필터 설정입니다. host_url_로 정확히 일치하는 진입점 1개만 ALLOWED 처리하고, 나머지는 도메인별로 세분화된 Blacklist/Whitelist로 제어합니다.
// C++ 테스트 설정 (url_filter_rules_unittest.cc) // ① host_url — 정확히 일치하면 blacklist/whitelist 검사 없이 바로 ALLOWED filter_.Init("https://www.mk.co.kr/audio-car", white, black); // ② Whitelist DomainPatternMap white = { {"www.mk.co.kr", {"@https://www.mk.co.kr/audio-car"}}, {"m.mk.co.kr", {"@https://m.mk.co.kr/audio-car"}}, {"account.mk.co.kr", {"@https://account.mk.co.kr/p/login/", "@https://account.mk.co.kr/p"}}, }; // ③ Blacklist DomainPatternMap black = { {"www.mk.co.kr", {"@https://www.mk.co.kr/premier", "@https://www.mk.co.kr/"}}, // 루트 prefix → 전체 차단 {"account.mk.co.kr", {"@https://account.mk.co.kr/p/join/", "@https://account.mk.co.kr/m/join/"}}, };
// "@" 패턴 → CompilePattern → kUrlPrefix 로 컴파일 // "@https://www.mk.co.kr/" (루트 URL) → subdomain_host = "www.mk.co.kr" 세팅 // → spec이 "https://www.mk.co.kr/"로 시작하는 모든 URL 매칭 (= www.mk.co.kr 전체 차단) // "@https://www.mk.co.kr/audio-car" (하위 경로) → subdomain_host = "" // → spec이 "https://www.mk.co.kr/audio-car"로 시작할 때만 매칭
🌐 www.mk.co.kr — Blacklist: @https://www.mk.co.kr/premier, @https://www.mk.co.kr/  ·  Whitelist: @https://www.mk.co.kr/audio-car
Host URL결과판단 근거
https://www.mk.co.kr/audio-car kUrlAllowed STEP1 host_url_ 정확 일치 → blacklist 검사 없이 바로 ALLOWED
https://www.mk.co.kr/audio-car/ kUrlBlocked Blacklist kUrlPrefix "https://www.mk.co.kr/" → spec이 해당 prefix로 시작 ✓
https://www.mk.co.kr/audio-car/article/123 kUrlBlocked Blacklist kUrlPrefix "https://www.mk.co.kr/" prefix 매칭 ✓
https://www.mk.co.kr/premier kUrlBlocked Blacklist kUrlPrefix "https://www.mk.co.kr/premier" prefix 매칭 ✓
https://www.mk.co.kr/premier/article kUrlBlocked Blacklist kUrlPrefix "https://www.mk.co.kr/premier" prefix 매칭 ✓
https://www.mk.co.kr/ kUrlBlocked Blacklist kUrlPrefix "https://www.mk.co.kr/" 정확 prefix 매칭 ✓
https://www.mk.co.kr/news kUrlBlocked Blacklist "https://www.mk.co.kr/" prefix 매칭 ✓ (루트 패턴이 모든 하위 경로 포함)

https://account.mk.co.kr/other kUrlBlocked Blacklist miss → Whitelist miss → BLOCKED
⚑ 핵심 포인트
  • @https://www.mk.co.kr/ (루트 URL 패턴)은 https://www.mk.co.kr/로 시작하는 모든 URL을 차단 → www.mk.co.kr 전체가 blacklist.
  • host_url_은 blacklist보다 먼저 체크되므로 https://www.mk.co.kr/audio-car 진입점만 안전하게 허용.
  • account.mk.co.kr에서 /p/join/은 blacklist가, /p 전체는 whitelist가 담당 → blacklist가 먼저 실행되므로 join 경로는 /p prefix에 포함되어도 차단됨.
